# js_runtime优化方案

针对 `js_runtime` 的缺点，我为你整理了一份详细的优化方案，涵盖了架构、内存、通信、调试等多个方面。

---

### 🧵 1. 优化线程模型：从“每实例一线程”到线程池

**缺点回顾**：每个 `JsRuntime` 实例创建一个专用操作系统线程，在实例数量较多时，线程创建和上下文切换的开销会成为瓶颈。

**优化方案**：引入**线程池 (Thread Pool)** 架构。

*   **设计思路**：不再为每个 `JsRuntime` 分配一个专属线程，而是维护一个全局的、可配置大小的线程池。所有 `JsRuntime` 实例共享这个线程池来执行 JS 代码。
*   **实现细节**：
    1.  **任务队列**：每个 `JsRuntime` 的 `eval` 等操作被封装成任务（Task），提交到一个全局的、多生产者单消费者（MPSC）或无锁队列中。
    2.  **工作线程 (Worker)**：线程池中的每个线程循环从队列中取出任务并执行。
    3.  **状态隔离**：虽然共享线程，但每个 `JsRuntime` 的 `Context` 仍然是独立的。任务在执行时，需要携带所属 `Runtime` 的 ID，以便工作线程能找到对应的 `Context` 来执行代码。
    4.  **配置化**：线程池的大小（如 `num_workers`）应作为 `JsRuntimeOptions` 的可配置项，让开发者根据应用场景（如 UI 密集型或计算密集型）进行调优。

### 🚀 2. 升级内存限制：从“估算”到“精确追踪”

**缺点回顾**：当前的内存限制仅基于代码和模块大小进行估算，无法精确追踪 `JsValue` 等运行时对象的实际内存占用。

**优化方案**：实现**基于自定义分配器 (Custom Allocator) 的精确内存追踪**。

*   **设计思路**：借鉴 QuickJS 的思路，在 Rust 层面实现一个自定义的全局分配器，它能感知并追踪每个 `JsRuntime` 实例的内存分配情况。
*   **实现细节**：
    1.  **实现 `GlobalAlloc`**：创建一个结构体 `TrackingAllocator`，实现 `std::alloc::GlobalAlloc` trait。
    2.  **关联 `RuntimeId`**：在分配时，通过某种方式（如线程局部存储）获取当前任务所属的 `RuntimeId`，并将分配的大小记录到该 `Runtime` 的专属计数器（`Arc<AtomicUsize>`）中。释放时则相应减少。
    3.  **检查与处理**：在每次分配前，检查计数器值加上请求大小是否超过预设的 `memory_limit`。若超出，则返回一个空指针（`std::ptr::null_mut()`），并在 Boa 的调用层将其转换为 JavaScript 的 `RangeError` 异常。
    4.  **配置入口**：在 `JsRuntimeOptions` 中保留 `memory_limit` 字段，其实现现在将真正生效。

### ⚡ 3. 改造双向通信：从“同步阻塞”到“异步非阻塞”

**缺点回顾**：JS 调用 Dart 回调（`dart_callback`）时是同步阻塞的，容易导致工作线程卡死。

**优化方案**：将 Dart 回调机制改造为**异步非阻塞模式**。

*   **设计思路**：当 JS 需要调用 Dart 函数时，立即返回一个 `Promise`，并将请求放入一个队列，由 Dart 侧的专用线程或通过 Flutter 的 `Isolate` 机制异步处理，处理完成后通过回调将结果传回 Rust，最终 resolve 那个 `Promise`。
*   **实现细节**：
    1.  **Promise 化**：在 Rust 层，为每个 `dart_callback` 调用创建一个 `Promise` 对象，并将其 `resolve` 函数与请求一起存储。
    2.  **消息队列**：将请求（包含参数、回调 ID）通过通道发送到 Dart 端。
    3.  **Dart 异步处理**：Dart 端在收到请求后，异步执行（例如，使用 `Future`）真正的业务逻辑。
    4.  **结果返回**：Dart 端执行完毕后，将结果通过另一个通道发送回 Rust。Rust 端根据回调 ID 找到对应的 `Promise` 并 `resolve`。
    5.  **超时机制**：为每个回调设置超时时间，避免因 Dart 端长时间无响应而泄漏资源。

### 🛠️ 4. 增强调试能力：完善错误信息与工具链

**缺点回顾**：跨语言调用导致错误堆栈不清晰，且缺少性能分析工具。

**优化方案**：建立一套**完整的错误传递与诊断体系**。

*   **实现细节**：
    1.  **结构化错误**：定义清晰的错误类型（如 `JsError`），不仅包含错误信息，还应包含 **JS 堆栈信息**（从 Boa 引擎获取）、**Rust 错误位置**（通过 `std::panic::Location`）和 **Dart 错误信息**（从 FRB 传递过来）。
    2.  **统一错误传递**：确保错误信息能在 Dart、Rust、JS 三层之间无损传递。利用 `flutter_rust_bridge` 的自定义类型映射能力，将 Rust 的 `JsError` 结构体完整映射到 Dart 侧。
    3.  **性能分析工具**：集成 Rust 的性能分析库（如 `puffin` 或 `tracing`），在关键路径（如 `eval`、`dart_callback`）添加 Span。同时，可以暴露 API 让 Dart 端能够启用或导出性能追踪数据，用于后续分析。

### 🧩 5. 解耦与扩展：轻量化 FRB 依赖并模块化扩展

**缺点回顾**：对 `flutter_rust_bridge` (FRB) 依赖过强，且扩展机制不灵活。

**优化方案**：

*   **解耦 FRB**：
    *   **抽象接口层**：在 Rust 侧定义一个清晰的、与 FRB 无关的 Core API 接口。FRB 生成的绑定代码只作为薄薄的一层适配器，将 Dart 调用转发给 Core API。
    *   **备选方案**：评估是否可以使用其他 FFI 方案（如 `dart-sys`）或手动编写一部分 FFI 代码，以减少对 FRB 特定版本的依赖。
*   **模块化扩展**：
    *   **定义标准 Extension Trait**：在 Rust 侧定义一个 `JsExtension` trait，包含 `register(global: &JsValue)` 等方法。
    *   **插件化加载**：允许通过配置文件或 API 动态加载扩展。将 `ConsoleExtension`、`FetchExtension` 等内置扩展与核心解耦，作为可选组件。
    *   **支持 Dart 侧扩展**：允许 Dart 侧通过注册函数的方式，向 JS 环境注入新的全局对象或函数，而无需修改 Rust 代码。

### 🏗️ 6. 升级引擎：评估并切换至更成熟的 JS 引擎

**缺点回顾**：Boa 引擎在性能和标准兼容性上不如 V8 等成熟引擎。

**优化方案**：这是一个长期且影响深远的决策，需要谨慎评估。

*   **短期方案**：持续关注 Boa 项目的进展，当其性能和兼容性满足需求时，升级 Boa 版本。
*   **长期方案**：评估将底层引擎替换为 **QuickJS** 或 **V8**。
    *   **QuickJS**：轻量、嵌入式友好，内存限制机制完善。Rust 绑定有 `rquickjs`，迁移成本相对较低。
    *   **V8**：性能最强、标准兼容性最好。但体积大，Rust 绑定（如 `deno_core`）相对复杂，且跨平台构建（尤其是移动端）挑战较大。
*   **策略**：建议在架构设计时将引擎层抽象出来（定义 `JSEngine` Trait），这样未来切换引擎时，只需实现新的 Trait，对上层 API 的影响可以降到最低。

### 📚 7. 完善文档与测试

**缺点回顾**：文档和示例不足，社区支持弱。

**优化方案**：

*   **API 文档**：使用 `cargo doc` 为 Rust 代码生成详细的文档，并为 Dart 代码添加完整的 dartdoc 注释。确保每个公开函数、类都有清晰的说明和使用示例。
*   **示例项目**：创建一个独立的 `example` 目录，包含从基础 `eval` 到复杂 `dart_callback` 通信的完整示例。
*   **单元测试与集成测试**：为 Rust 核心逻辑编写单元测试，并为 Dart API 编写集成测试，确保在多个平台上都能正常工作。这不仅能提高代码质量，也是最好的“文档”。
*   **贡献指南**：编写 `CONTRIBUTING.md`，降低新开发者的参与门槛。

---

### 💎 总结

这份优化方案是系统性的，旨在将 `js_runtime` 从一个功能原型打造成一个生产就绪的、高性能且可靠的 JavaScript 运行时。你可以根据项目的实际需求和优先级，分阶段实施这些建议。
